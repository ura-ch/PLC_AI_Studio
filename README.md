## 1. Что это и для кого

PLC AI Studio — это рабочий инструмент инженера АСУ ТП. Он берёт на себя самую трудоёмкую часть пусконаладки и проектирования — написание прикладной программы контроллера по техническому заданию — и доводит результат до **компилируемого, проверенного кода**, а не до «черновика из чата».

Программа предназначена для:

- инженеров-программистов ПЛК, которым нужно быстро получить корректный каркас программы по большому перечню сигналов;
- проектировщиков АСУ ТП, которым важны схемы (SFC/LD/FBD), мнемосхема и пояснительная записка;
- наладчиков, которым нужен понятный, проверяемый и переносимый код.

Программа работает **локально на вашем компьютере** (Windows). Облако нужно только если вы пользуетесь облачными моделями (DeepSeek, OpenAI и т. п.); при использовании локальной модели (Ollama) интернет не требуется вовсе. Ключи доступа к моделям хранятся только у вас.

**Важно понимать ограничение ответственности.** Сгенерированный код — это качественный черновик, требующий проверки и сертификации инженером. Финальное решение и ответственность всегда остаются за специалистом.

```mermaid
        U2["Редактор кода ST"]
        U3["Редакторы схем: SFC / LD / FBD / SCADA"]
        U4["Панель сравнения ИИ"]
    end

    subgraph CORE["Ядро приложения"]
        R1["Приём и разбор файлов"]
        R2["Сборка задания для ИИ"]
        R3["Проверка и исправление кода"]
        R4["Экспорт результатов"]
    end

    subgraph AI["Движки ИИ"]
        A1["DeepSeek / OpenAI / Anthropic / Google"]
        A2["Локальная модель Ollama"]
        A3["Агенты: R2-PLCGen, Agents4PLC, truST"]
    end

    subgraph TOOLS["Инженерный инструментарий"]
        T1["Каркас из IOLIST"]
        T2["Нормализатор под цель"]
        T3["Гейт компиляции matiec / CODESYS"]
        T4["Оценка и выбор лучшего"]
    end

    subgraph DATA["Данные и память"]
        D1["Библиотека эталонов (RAG)"]
        D2["Базы сценариев"]
        D3["Ваши библиотеки ФБ"]
    end

    UI --> CORE
    CORE --> AI
    CORE --> TOOLS
    AI --> TOOLS
    TOOLS --> DATA
    AI --> DATA
    CORE --> UI
```

*Схема 1. Общее устройство программы. Интерфейс обращается к ядру; ядро использует движки ИИ и инженерный инструментарий; всё опирается на данные и память.*

Коротко о слоях:

- **Интерфейс** — то, что вы видите: загрузка файлов, редактор кода, редакторы схем, сравнение результатов разных ИИ.
- **Ядро** — принимает файлы, готовит задание для ИИ, организует проверку и исправление, делает экспорт.
- **Движки ИИ** — собственно «авторы» прикладной логики: обычные модели (быстро) и агенты (умнее, но дольше).
- **Инженерный инструментарий** — то, что делает результат **инженерным, а не «как получилось у чата»**: каркас из IOLIST, нормализатор, гейт компиляции, оценка.
- **Данные и память** — проверенные эталоны, базы сценариев, ваши собственные библиотеки блоков.

## 4. Главная идея: почему код получается рабочим

Раньше типичный подход был такой: «дай нейросети ТЗ и список из сотен сигналов — пусть напишет всю программу за один раз». На практике это давало неполный охват сигналов, повторно объявленные переменные, недопустимые конструкции и обрывы на середине.

PLC AI Studio устроен иначе. Ключевой принцип:

> **Всё, что можно посчитать механически, считает сама программа, а нейросети оставляют только то, что требует «соображения» — прикладную логику небольших блоков.**

Что это значит на практике:

- **Имена переменных, типы, адреса, экземпляры оборудования и разводку входов/выходов программа строит сама** из IOLIST — детерминированно, без участия ИИ. Это убирает целые классы ошибок «как явление»: не бывает ни пропущенных сигналов, ни дублей, ни «выдуманных» имён.
- **Нейросеть пишет только тело функциональных блоков** — короткую, осмысленную логику («как запускать мотор», «как обработать аварию»). Это маленькая задача, с которой модель справляется надёжно и без обрывов.
- **Результат собирается обратно автоматически**, приводится к нужному диалекту и прогоняется через цикл компиляции до состояния «без ошибок».

Именно поэтому код получается полным, корректным по структуре и проверяемым.

### 5.3. Каркас из IOLIST

Это «фундамент», который программа строит **сама, без ИИ**:

- **Канонические имена** — корректные идентификаторы по стандарту с префиксом по типу (`x` — логический, `i` — целый, `r` — вещественный, `w` — слово). Имена с точками и спецсимволами очищаются. Совпадения проверяются без учёта регистра (как в стандарте), поэтому коллизий имён не возникает.
- **Адреса по типу** — логические сигналы получают `%IX/%QX`, целые/слова — `%IW/%QW`, вещественные — `%ID/%QD`; входы и выходы определяются по направлению.
- **Распознавание оборудования** — по префиксу тега (`CNV01_…`, `ST02_…`) сигналы группируются в экземпляры устройств.
- **Один функциональный блок на тип устройства** — все одинаковые по набору сигналов устройства получают общий ФБ, а разные — разные. Например, станки и моечная станция одной линии автоматически попадают в разные блоки.
- **Создание экземпляров и разводка** — для каждого устройства объявляется экземпляр блока и автоматически подключаются его входы и выходы к нужным сигналам.

Главный эффект: **100% охват сигналов гарантирован** независимо от поведения нейросети.

### 5.4. Генерация логики (нейросеть пишет тела блоков)

Программа отдаёт модели готовый каркас и просит заполнить **только логику тел блоков** — что именно делает «мотор», «клапан», «горелка». Объявления и разводку модель не трогает. В задание подмешиваются проверенные **эталоны по категории** (раздел 9) — модель видит образцы «как надо».

### 5.5. ФБ-чанкинг для больших проектов

Если проект большой (много сигналов, несколько типов блоков), программа не пытается получить всё одним огромным ответом — это и приводило к обрывам. Вместо этого включается **ФБ-чанкинг**: тело **каждого** блока запрашивается отдельным коротким запросом.

```mermaid
flowchart LR
    A["Каркас:<br/>5 типов блоков + Main"] --> B1["Запрос 1:<br/>тело FB_CNV"]
    A --> B2["Запрос 2:<br/>тело FB_ST"]
    A --> B3["Запрос 3:<br/>тело FB_RB"]
    A --> B4["…"]
    A --> B5["Запрос N:<br/>координация Main"]
    B1 --> C["Сборка программы целиком"]
    B2 --> C
    B3 --> C
    B4 --> C
    B5 --> C
    C --> E["Плюс описание и SFC<br/>отдельным коротким запросом"]
```

*Схема 3. ФБ-чанкинг: вместо одного гигантского (и обрывающегося) ответа — несколько коротких, которые гарантированно укладываются в лимит, а затем собираются в одну программу.*

Так результат для крупных проектов получается **полным**, без обрезанного кода. Этот режим работает со всеми обычными моделями (DeepSeek, OpenAI, Anthropic, Google, Ollama).

### 5.6. Сборка программы

Полученные тела блоков программа вставляет в заранее построенный каркас. Сборка детерминированная: одни и те же исходные данные всегда дают один и тот же корректный каркас. Если модель по какой-то причине вернула не тела, а целую программу — срабатывает безопасный запасной путь, и хуже не становится.

- для **CODESYS** — бережно, с сохранением удобных расширений CODESYS.

### 5.8. Гейт компиляции

Вместо «проверили один раз и надеемся» работает **замкнутый цикл**:

```mermaid
flowchart LR
    A["Код"] --> B["Нормализация<br/>под цель"]
    B --> C{"Проверка цели<br/>matiec / CODESYS"}
    C -->|"Ошибок нет"| OK["Готово"]
    C -->|"Есть ошибки"| F["Точечное исправление"]
    F --> B
    C -.->|"нет прогресса"| STOP["Стоп: защита от зацикливания"]
```

*Схема 4. Гейт компиляции: цикл «нормализовать → проверить → исправить» повторяется до состояния без ошибок, со страховкой от бесконечного цикла.*

### 5.9. Оценка вариантов и выбор лучшего (В режиме Строгая генерация)

Если генерация шла в нескольких ИИ сразу, каждый вариант **оценивается автоматически** (компилируется ли, насколько полно покрыты сигналы, насколько код модульный). Программа сама подсказывает лучший вариант значком, а не заставляет вас вручную сравнивать «сломанные» результаты.

## 6. Режимы работы

Все три режима ведут к одному и тому же качественному конвейеру (раздел 5). Отличается только то, **как вы подаёте задачу**.

```mermaid
flowchart TB
    M1["Ручной режим"] --> CORE["Единый конвейер генерации<br/>каркас - логика - сборка - нормализация - гейт - выбор лучшего"]
    M2["Блок-объект"] --> CORE
    M3["Блок-проект: комплекс"] --> CORE
    CORE --> OUT["Код, схемы, описание, отчёт"]
```

*Схема 5. Любой режим использует одно и то же проверенное ядро генерации.*

### 6.1. Ручной режим

Основной рабочий режим: вы выбираете категорию (тип установки), загружаете ТЗ и IOLIST и запускаете генерацию. Можно запустить **несколько ИИ параллельно** и сравнить, либо использовать режим повышенной надёжности с пошаговой проверкой по каждому блоку.

![1782053507699](markdown_files/image/Program_description/1782053507699.png)
*Рис. 6. Ручной режим: выбор сценария, загрузка ТЗ и IOLIST.*

### 6.2. Блок-объект

Режим для **одного объекта**, оформленного как структурированная система. Удобно, когда нужен один аккуратный объект (например, одна насосная станция) с его сигналами и логикой.

![1782053531042](markdown_files/image/Program_description/1782053531042.png)
*Рис. 7. Режим «Блок-объект»: мастер одиночного объекта.*

### 6.3. Блок-проект (комплекс)

Режим для **комплекса из нескольких систем**, связанных между собой (например, котельная + водоподготовка + пожарная сигнализация с общими сигналами вроде «Пожар», «Тепло готово»).

```mermaid
flowchart LR
    A["Загрузка ZIP<br/>несколько систем"] --> B["Разбор подсистем<br/>и их типов"]
    B --> C["Определение порядка<br/>и межсистемных сигналов"]
    C --> D["Генерация каждой системы<br/>тем же конвейером"]
    D --> E["Совместная проверка<br/>сценарий взаимодействия"]
    E --> F["Итоговый архив:<br/>код и пояснительная записка"]
```

*Схема 6. Этапы режима «Блок-проект». Каждая система проходит тот же конвейер, что и одиночный объект.*

## 7. Две цели проверки: matiec и CODESYS

Один и тот же код можно проверять и приводить под **две независимые цели**. Это важно, потому что «правильность» зависит от того, где код будет работать.

| Цель          | Что это                                                                                                                                                         | Когда выбирать                                                                                                                          |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **matiec**  | Бесплатный компилятор «по букве стандарта» МЭК 61131-3. Самый строгий эталон переносимости.   | Когда нужен максимально переносимый, стандартный код, либо для строгой проверки. |
| **CODESYS** | Среда со своими удобными расширениями (битовый доступ, комментарии `//`, незанятые выходы). | Когда целевой контроллер программируется в CODESYS и расширения допустимы.                 |

Принципы, которые стоит знать:

* Код, чистый для **matiec**, будет корректен и в **CODESYS** (стандарт — подмножество).
* Кнопка проверки позволяет выбрать цель; кнопка исправления приводит код под выбранную цель.

* По умолчанию результат остаётся в стиле CODESYS; вариант для matiec получается по запросу (проверить под matiec → исправить)

![1782053954769](markdown_files/image/Program_description/1782053954769.png)

![1782053980818](markdown_files/image/Program_description/1782053980818.png)

*Рис. 9. Выбор цели проверки и результат: ноль ошибок либо список замечаний компилятора.*

> **Про matiec на вашем компьютере.** Если рядом с программой лежит компилятор `iec2c` (в папке `bin/`), проверка использует **настоящую компиляцию**. Если его нет — этап нативной компиляции пропускается, остаётся встроенная проверка; программа при этом продолжает работать.

## 9. Библиотека эталонов: на чём «заземляется» генерация

Эталоны делятся на **общие строительные блоки** (применимы во многих сценариях) и **профильные** (специфичны для конкретной отрасли). Сейчас их **19** на **9 категорий**.

## 13. Проверка безопасности и трассируемость требований

Компиляция (раздел 5.8) отвечает на вопрос «синтаксис верный?», семантический анализатор (раздел 12) — «нет ли опасных конструкций?». Но ни то, ни другое не проверяет **главное для АСУ ТП**: *делает ли логика то, что требует ТЗ* — сработает ли защита, не запустится ли механизм при аварии. Этот слой закрывает именно этот вопрос: программа **прогоняет сгенерированный код на встроенном соф-ПЛК** по сценариям безопасности и проверяет инварианты — правила, которые обязаны выполняться всегда.

```mermaid
flowchart TB
    GEN["Сгенерированный код"] --> SIM["Прогон на соф-ПЛК<br/>по сценариям категории"]
    TZ["ТЗ: контракт связей<br/>и критические запреты"] --> RULES["Проверяемые требования"]
    CAT["Библиотека инвариантов<br/>по категории установки"] --> RULES
    RULES --> SIM
    SIM --> VERDICT{"Инварианты<br/>выполнены?"}
    VERDICT -->|"да"| OK["✅ Безопасно"]
    VERDICT -->|"нет"| BAD["⛔ Нарушение защиты/блокировки"]
    OK --> MATRIX["Единая матрица<br/>трассируемости"]
    BAD --> MATRIX
```

*Схема 7. Логика проверки безопасности: код исполняется на модельном ПЛК, требования берутся из библиотеки категории и из ТЗ, итог сводится в одну матрицу.*

![1782057159338](markdown_files/image/Program_description/1782057159338.png)

### 13.1. Поведенческая проверка по категории

Для каждой категории установки есть **библиотека инвариантов безопасности** — правил вида «такого быть не должно никогда». Правило написано в *ролях* (например, «E-stop», «команда пуска», «привод»), а программа сама находит эти роли среди ваших сигналов по типу (DI/DO/AI/AO) и описанию и подставляет реальные теги. Если нужной роли в проекте нет — правило просто пропускается, лишнего не штрафует.

Программа подаёт на код «опасную» ситуацию и смотрит на реакцию. Примеры встроенных проверок:

| Категория                              | Что проверяется (инвариант безопасности)                                                                                                 |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Конвейеры                              | Привод не включается и снимается при активном E-stop; снимается по неисправности/перегрузу          |
| Насосные / водоподготовка | Насос не работает при «сухом ходе» (нет воды)                                                                                           |
| Вентиляция (BMS/HVAC)                 | При низкой температуре включается защита от замораживания, заслонка закрывается                       |
| Котлы                                      | При пожаре закрывается газовый/топливный клапан; нагрев отключается при превышении давления |
| Пожарная сигнализация       | Срабатывание извещателя поднимает оповещение                                                                                       |
| Энергетика                            | Выключатель отключается по срабатыванию защиты (РЗА)                                                                           |
| Движение / приводы               | На концевике движение запрещено; E-stop снимает разрешение привода                                                      |

Эти правила — **стартовый набор**, который настраивается под объект (раздел 13.5). Полярность сигналов (например, NC-контакт E-stop, где TRUE = норма) и пороги (давление, температура) вы при необходимости поправляете под конкретный проект.

### 13.2. Влияние на выбор лучшего варианта

Поведенческая проверка встроена в автоматическую оценку вариантов (раздел 5.9). К прежним критериям (компилируемость, охват сигналов, модульность) добавляется **поведение**, причём с жёстким приоритетом безопасности:

- вариант получает прибавку за пройденные проверки;
- **любое нарушение инварианта безопасности резко опускает балл** — небезопасный, но «красиво скомпилированный» код не может победить безопасный с косметическими замечаниями.

В ручном режиме вердикт виден прямо на вкладке каждого ИИ: «✅ безопасно N/N» или «⛔ небезопасно» с указанием нарушенного правила.

### 13.3. Согласие провайдеров (ensemble)

Если генерация шла в нескольких ИИ, программа прогоняет **одни и те же** проверки на коде каждого и сравнивает вердикты. Если все реализовали блокировку одинаково — доверие выше. Если по какому-то правилу **расхождение** (часть моделей прошла, часть нет) — это место отмечается флагом «нужен глаз инженера».

Важно: большинство — не истина. Согласие используется как **подсказка и мягкий ориентир**, оно не отменяет проверку безопасности: одинокая, но правильная модель не штрафуется.

### 13.4. Контракт ТЗ → проверяемые требования

В режиме «Блок-проект» из ТЗ извлекаются дополнительные требования:

- **Межсистемные блокировки.** Контракт связей (раздел 6.3) задаёт, какие сигналы система-потребитель обязана читать (например, «Котёл» читает «Пожар»). Программа проверяет, что каждый такой сигнал **реально используется в коде** потребителя. Если нет — блокировка из ТЗ объявлена, но не реализована; это подсвечивается при сохранении системы.
- **Критические запреты.** Запреты из ТЗ («нельзя запускать насос при сухом ходе», «при пожаре закрыть газовый клапан») сопоставляются с известными паттернами и превращаются в проверяемые инварианты. Что распознано — проверяется автоматически; что **не** распознано — возвращается списком инженеру (программа не выдумывает требований из текста, который не поняла однозначно).

### 13.5. Единая матрица трассируемости

Все требования сводятся в **одну таблицу** — чтобы не было параллельных списков. Каждая строка помечена источником и статусом, и связана с кодом:

| Источник строки | Откуда                                                     | Возможный статус                                                         |
| ----------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| ТЗ                          | требования из брифа и IOLIST                   | реализовано / доказано / отсутствует                      |
| safety / проверка     | инварианты безопасности категории | **доказано** (прогон пройден) / **провалено** |
| ТЗ-запрет             | запреты из ТЗ                                         | доказано / провалено                                                   |
| контракт              | межсистемные блокировки                    | реализовано /**отсутствует**                                |

«Доказано» означает не «скомпилировалось», а «проверено исполнением на модельном ПЛК». В режиме «Блок-объект» матрица показывается в панели Уровня 1; клик по строке подсвечивает соответствующие строки кода. В «Блок-проекте» такая матрица формируется по каждой подсистеме.

> Реконструкция требований «из существующего кода» (для анализа чужих, ранее написанных программ — brownfield) остаётся отдельным инструментом и намеренно не смешивается с этой матрицей: это разные по смыслу вещи — *предписанное требование* против *восстановленного по факту*.

### 13.6. Где настраивать

Встроенные инварианты есть во всех категориях. Свои правила под конкретный объект добавляются файлом `behavior_rules/<категория>.json` (формат и примеры — в `behavior_rules/README.md`, см. раздел 14 о структуре папок). Так инженер расширяет проверки, не трогая код программы.

### 13.7. Как это работает в трёх режимах

| Режим            | Что даёт проверка безопасности                                                                                                                                                                                                         |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ручной          | Вердикт «безопасно / небезопасно» и ensemble-расхождения на вкладках ИИ; влияет на выбор лучшего                                                                                         |
| Блок-объект | Полная панель: прогон, матрица трассируемости (требования ТЗ + проверки безопасности в одной таблице), подсветка кода                                            |
| Блок-проект | Каждая система генерируется ручным потоком (те же вердикты) + проверка межсистемных блокировок контракта и запретов ТЗ по каждой подсистеме |

## 15. Частые вопросы

**Нужен ли интернет?**
Для облачных моделей — да. Для локальной модели (Ollama) — нет. Остальные функции работают без сети.

**Какие форматы IOLIST поддерживаются?**
`.xlsx`, `.xls`, `.xlsm`, `.csv`, `.tsv`, `.json`. Строка заголовков находится автоматически.

**Куда уходят мои API-ключи?**
Только в локальное хранилище на вашем компьютере, напрямую к провайдеру. В программе и в логах ключей нет.

**Почему один из ИИ «не вернул код» или код оказался короче?**
Чаще всего это упирание в лимит длины ответа на очень больших проектах. Для обычных моделей включается ФБ-чанкинг (короткие запросы), а для больших перечней автоматически поднимается лимит ответа. Если ваша модель не принимает большой лимит, его можно ограничить настройкой `MAX_OUTPUT_TOKENS_CAP` (например, значением 24000).

**Почему вместо Ollama показана конкретная модель (qwen2.5-coder и т. п.)?**
Это нормально: Ollama запускает установленную у вас локальную модель и показывает её настоящее имя (см. раздел 8.1).

**Откуда в памяти много записей, если я столько проектов не делал?**
Это встроенные проверенные эталоны (раздел 9). Они нужны для качества и не мешают вашим данным.

**Можно ли добавить свои функциональные блоки?**
Да — положите `.st`-файлы в `user_library/` (раздел 14), и программа будет предлагать модели использовать именно их.

**Заменяет ли программа инженера?**
Нет. Результат — качественный черновик, требующий проверки и сертификации. Ответственность остаётся за специалистом.

**В каком стиле выдаётся код по умолчанию — matiec или CODESYS?**
По умолчанию — в стиле CODESYS. Переносимый вариант под matiec получается по запросу через проверку и исправление под эту цель.

## 16. Полный глоссарий

- **АСУ ТП** — автоматизированная система управления технологическим процессом.
- **ПЛК** — программируемый логический контроллер.
- **МЭК 61131-3 (IEC 61131-3)** — международный стандарт языков программирования ПЛК.
- **ST (Structured Text)** — текстовый язык программирования ПЛК (паскалеподобный).
- **POU (Program Organization Unit)** — программный модуль: `PROGRAM`, `FUNCTION_BLOCK`, `FUNCTION`.
- **FUNCTION_BLOCK (ФБ)** — переиспользуемый модуль логики со своими входами/выходами и внутренней памятью.
- **IOLIST** — перечень входов/выходов (сигналов) объекта.
- **Тег** — отдельный сигнал в IOLIST.
- **NC/NO** — нормально замкнутый / нормально разомкнутый контакт.
- **Уставка** — заданное значение параметра (порог, время, давление и т. п.).
- **SFC (Sequential Function Chart)** — графический язык диаграмм последовательности.
- **LD (Ladder Diagram)** — релейно-контактные схемы.
- **FBD (Function Block Diagram)** — схема функциональных блоков.
- **SCADA** — система диспетчеризации и отображения технологического процесса (здесь — генерация мнемосхемы).
- **matiec** — открытый компилятор стандартного МЭК 61131-3; `iec2c` — его исполняемый файл.
- **CODESYS** — среда разработки ПЛК со своими расширениями языка.
- **PLCopen XML** — обменный формат проектов между средами разработки ПЛК.
- **Каркас (scaffold)** — детерминированно построенная из IOLIST основа программы (объявления, адреса, экземпляры, разводка).
- **ФБ-чанкинг** — генерация тела каждого функционального блока отдельным запросом с последующей сборкой.
- **Нормализатор** — механическое приведение кода к выбранному диалекту (matiec/CODESYS).
- **Гейт компиляции** — цикл «нормализовать → проверить → исправить» до отсутствия ошибок.
- **Best-of-N** — автоматическая оценка нескольких вариантов и выбор лучшего.
- **RAG / эталоны** — заранее проверенные образцы кода, показываемые модели как примеры для повышения качества.
- **Агент** — многошаговый ИИ-пайплайн (план → генерация → проверка → исправление).
